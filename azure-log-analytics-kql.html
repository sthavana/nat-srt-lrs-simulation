Azure Log Analytics & KQL — Deep Dive

A from-scratch guide to Azure Log Analytics and the Kusto Query Language (KQL) — the tools used to investigate Harmonic LRS stream health.

Every concept is taught generically (the skill transfers to any Log Analytics table), then immediately grounded in a real LRS example against the DatapathStats_CL table. Green = generic skill; amber = LRS-grounded example.

Once you're comfortable here, the companion LRS Troubleshooting Guide walks a real investigation end to end. ▶ LRS Troubleshooting Guide — the 10-query Globo investigation
1 — Azure Log Analytics: Orientation
Before any query, it helps to know where you are. Azure Log Analytics is the query layer over a Log Analytics workspace — a container that stores log and metric data in tables. You write KQL in the Logs blade and get back a results grid (or a chart).
ConceptWhat it is
WorkspaceThe data container. For LRS this is HarmonicLRSLogAnalytics. Everything you query lives inside one workspace.
TableA named, schema'd stream of rows (e.g. DatapathStats_CL). The _CL suffix means a Custom Log table; _s/_d/_t/_b column suffixes mean string/real/datetime/bool.
Logs bladeThe query editor in the Azure portal. Paste KQL, press Run (or Shift+Enter).
ScopeWhat the query runs against — a workspace, a resource, or a resource group. Set it at the top-left before running.
Time rangeThe portal time picker. It adds an implicit time filter on TimeGenerated. If your query has its own where TimeGenerated between (...), set the picker to Set in query so the two don't fight.
Results vs ChartOutput shows as a table by default. Add | render timechart (or switch the Chart tab) to visualise.
Save / PinSave a query to the workspace for reuse, or pin a chart to an Azure dashboard for monitoring.
The time-picker gotcha. A query can return "no results" simply because the portal time range is narrower than your where TimeGenerated clause — or wider, masking it. When a query has an explicit time filter, set the picker to Set in query. Log Analytics also has a retention limit: data older than the workspace's retention period is gone.
2 — The KQL Mental Model: a Pipeline of Tables
KQL reads top to bottom, left to right. You start with a table and pass it through a series of operators, each separated by a pipe |. Every operator takes a table in and returns a table out — so the output of one step is the input to the next. There is no SELECT … FROM … WHERE ordering to memorise; you just describe the transformation in the order it happens.
DatapathStats_CL
Start with the whole table (every row, every column)
▼
| where …
Keep only the rows you care about (fewer rows)
▼
| summarize … by …
Aggregate rows into groups (fewer rows, new columns)
▼
| extend …
Add a calculated column (e.g. a percentage)
▼
| sort by …
Order the result, then render or read it
Read it as a sentence: "Take DatapathStats_CL, keep Globo rows in the match window, total the dropped packets per stream, work out the drop percentage, and sort worst-first." That sentence is the query — in that order.
3 — Filtering Rows: where
where Keep only matching rows Generic
where filters the table down to rows where a condition is true. Chain multiple wheres (or combine with and/or) — each one runs on the output of the previous. Filter early and filter hard: the less data flows downstream, the faster and cheaper the query.
SomeTable | where Status == "error" | where Region in ("eu-west", "us-east") | where Timestamp > ago(1h)
Grounded in DatapathStats_CL
// Globo's ingress rows in the match window only DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | where recv_pkt_dropped_d > 0
String matching — pick the right operator:
OperatorMeaningExample
==Exact, case-sensitive equalityname_s == "LL06 - PSC"
=~Exact, case-insensitiveregion_s =~ "SOUTHAMERICA-EAST1-B"
hasWhole-word match (fast, indexed)name_s has "LL06"
containsSubstring match (slower, not indexed)id_s contains "fa6446"
inMatches any value in a listname_s in ("LL06 - PSC","LL07 - PSC")
startswithPrefix matchid_s startswith "ingress-"
Prefer has over contains when matching whole words — it uses the term index and is much faster on large tables.
4 — Shaping Columns: project & extend
project Choose / rename / reorder columns Generic
project keeps only the columns you list (and sets their order). project-away does the inverse — drop a few, keep the rest. Use it to cut a wide table down to a readable result.
SomeTable | project Time = Timestamp, User, Action // keep 3 cols, rename one SomeTable | project-away RawData, _ResourceId // drop noise, keep the rest
Grounded in DatapathStats_CL
DatapathStats_CL | where name_s has "LL06" | project TimeGenerated, name_s, upstream_role_s, recv_pkt_loss_d, recv_pkt_dropped_d, recv_bitrate_d
extend Add a calculated column Generic
extend appends a new column computed from existing ones — arithmetic, string functions, conditionals. The original columns stay. This is how you turn raw counters into a meaningful metric like a percentage.
SomeTable | extend DurationSec = DurationMs / 1000.0
Grounded in DatapathStats_CL — counts to a percentage
DatapathStats_CL | where account_prefix_s has "globocomuni" | summarize Recv = sum(recv_pkt_d), RecvDropped = sum(recv_pkt_dropped_d) by name_s | extend RecvDropPct = round(100.0 * RecvDropped / Recv, 4)
Watch integer division. Multiply by 100.0 (not 100) to force floating-point math — otherwise 5 / 1000 returns 0. round(x, 4) trims to 4 decimal places.
5 — Aggregation: summarize … by
This is the heart of KQL. summarize collapses many rows into one row per group, computing aggregate values. The by clause names the grouping key(s). Without by, you get a single total row.
summarize Group rows and compute aggregates Generic
Pattern: summarize <NewCol> = <agg>(<col>), … by <group cols>. Common aggregate functions:
FunctionReturns
sum(x)Total of x across the group
avg(x)Mean of x
max(x) / min(x)Largest / smallest value
count()Number of rows in the group
dcount(x)Distinct (unique) count of x
countif(pred)Count of rows where a condition is true
arg_max(x, *)The whole row where x is largest
Grounded in DatapathStats_CL — per-stream health totals
DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize Sent = sum(snd_pkt_d), Recv = sum(recv_pkt_d), RecvDropped = sum(recv_pkt_dropped_d), Samples = count(), Legs = dcount(socket_d) by name_s
name_s Sent Recv RecvDropped Samples Legs LL06 - PSC 51,200,400 51,180,900 23,267 600 2 LL07 - PSC 50,990,100 50,975,800 14,056 600 2 LL08 - PSC 51,010,700 51,010,700 0 600 2
Group by more than one key. by name_s, upstream_role_s gives one row per leg per stream — the grouping that separates primary from backup. by bin(TimeGenerated, 1m), name_s groups into per-minute, per-stream buckets (next section).
6 — Time Series: bin(), ago() & render
bin() Bucket a timestamp into fixed intervals Generic
bin(TimeGenerated, 1m) rounds each timestamp down to the nearest minute, so rows in the same minute share a value — the standard way to build a time series with summarize. Use 5m, 1h, 1d etc. for coarser buckets. Time literals: datetime(2026-06-11T18:25:00Z) for absolute, ago(2h) / now() for relative.
SomeTable | where Timestamp > ago(6h) | summarize Errors = count() by bin(Timestamp, 5m) | render timechart
Grounded in DatapathStats_CL — drops per minute, per stream
DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize Dropped = sum(recv_pkt_dropped_d) by bin(TimeGenerated, 1m), name_s | render timechart
Why this matters for LRS: binning by minute is exactly how the Globo investigation found the drops were confined to a 3-minute window (18:25–18:28 UTC) instead of spread across the match — the single result that killed the "match-load" theory.
render Visualise the result Generic
render turns a result into a chart. timechart expects a datetime first column; columnchart/barchart/piechart suit categorical data. When you summarize … by bin(t,…), Category, each category becomes its own line.
// Bitrate per stream over time — one line per stream DatapathStats_CL | where account_prefix_s has "globocomuni" | summarize AvgBitrate = avg(recv_bitrate_d) by bin(TimeGenerated, 1m), name_s | render timechart
7 — Ordering & Limiting: sort, top, take
sort / top / take Order the result and cap the rows Generic
OperatorWhat it does
sort by X descOrder rows by X (default is descending; add asc for ascending)
top 10 by XSort by X and keep the first 10 — "worst/biggest N"
take 100Grab any 100 rows (no order guarantee) — for a quick peek at a table
distinct A, BUnique combinations of the listed columns
Grounded in DatapathStats_CL — worst streams first, and what they share
// Rank streams by unrecovered drops DatapathStats_CL | where account_prefix_s has "globocomuni" | summarize RecvDropped = sum(recv_pkt_dropped_d) by name_s | top 5 by RecvDropped desc // What do the affected streams have in common? DatapathStats_CL | where name_s in ("LL06 - PSC", "LL07 - PSC", "LL08 - PSC") | distinct name_s, region_s, upstream_role_s, socket_d
8 — Power Tools: let, conditionals, getschema
let Name a value, expression, or sub-table Generic
let declares a reusable binding at the top of a query — a scalar, a list, or even a whole sub-query. It removes repetition (one place to change the account or time window) and makes long queries readable.
Grounded in DatapathStats_CL — parameterise the investigation
let acct = "globocomuni"; let tStart = datetime(2026-06-11T18:24:00Z); let tEnd = datetime(2026-06-11T18:30:00Z); let streams = dynamic(["LL06 - PSC", "LL07 - PSC", "LL08 - PSC"]); DatapathStats_CL | where account_prefix_s has acct | where TimeGenerated between (tStart .. tEnd) | where name_s in (streams) | project TimeGenerated, name_s, upstream_role_s, recv_pkt_dropped_d
iff() / case() Conditional (computed) columns Generic
iff(condition, ifTrue, ifFalse) is an inline if. case(c1, v1, c2, v2, …, default) handles multiple branches. Pair with extend to label rows.
Grounded in DatapathStats_CL — flag each leg's health
DatapathStats_CL | where name_s has "LL06" | extend Health = case( recv_pkt_dropped_d > 0, "DROPS", recv_pkt_loss_d > 0, "loss (recovered)", "clean") | project TimeGenerated, upstream_role_s, recv_bitrate_d, Health
getschema Discover a table's columns LRS-handy
When a field name errors (or you're new to a table), getschema lists every column and its type. The fastest way to confirm an LRS field name and its _s/_d suffix.
DatapathStats_CL | getschema | project ColumnName, ColumnType | sort by ColumnName asc
9 — Combining Tables: join & union (brief)
join / union Correlate or stack multiple tables Generic
join matches rows from two tables on a shared key (like SQL joins: inner, leftouter, etc.). union stacks rows from several tables/queries into one result. For LRS, the DatapathStats_CL table is usually self-sufficient — but a self-join can line up a stream's two legs side by side, and union is useful if stats land in more than one table.
Grounded in DatapathStats_CL — baseline vs. window via a self-join
let windowStats = DatapathStats_CL | where TimeGenerated between (datetime(2026-06-11T18:24:00Z) .. datetime(2026-06-11T18:30:00Z)) | summarize WindowDropped = sum(recv_pkt_dropped_d) by name_s; DatapathStats_CL | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize TotalDropped = sum(recv_pkt_dropped_d) by name_s | join kind=inner windowStats on name_s | extend PctInWindow = round(100.0 * WindowDropped / TotalDropped, 1)
Reads as: total drops per stream over the whole evening, joined to drops in the tight window — PctInWindow then shows what fraction of all drops happened in those 3 minutes (near 100% for LL06/LL07).
10 — Putting It Together
Every operator above, in one query — the "run this first" health summary from the troubleshooting guide. Read it top to bottom as a sentence:
let acct = "globocomuni"; DatapathStats_CL // 1. start with the table | where account_prefix_s has acct // 2. filter to the customer | where TimeGenerated > ago(7d) // 3. filter to recent data | summarize // 4. aggregate per stream Sent = sum(snd_pkt_d), Recv = sum(recv_pkt_d), SendDropped = sum(snd_pkt_dropped_d), RecvDropped = sum(recv_pkt_dropped_d) by name_s | extend // 5. derive percentages RecvDropPct = round(100.0 * RecvDropped / Recv, 4), SendDropPct = round(100.0 * SendDropped / Sent, 4) | sort by RecvDropPct desc // 6. worst first
