LRS via Private Service Connect — Loopback Setup Guide

This guide details how to connect SRT sources in a customer GCP VPC to Harmonic LRS Cloud Source using Google Private Service Connect (PSC) — Endpoint model with the Loopback approach.

How the loopback approach differs from iptables: Instead of iptables DNAT/SNAT rules on the LRS VM, the ILB frontend IP is added to the VM's loopback interface. The SRT listener is then bound directly to that IP — so response packets naturally carry src = ILB IP, satisfying PSC NAT session tracking without any iptables rules. This requires Syndicate to pass the ILB IP as the SRT listener bind address.

Architecture context: LRS runs in its own dedicated Harmonic VPC. Harmonic publishes each LRS Cloud Source via an ILB + PSC Service Attachment. The customer creates a PSC Endpoint in their own VPC and SRT encoders connect to that private IP.

Prerequisite: SRT streams from the customer headend must already be available in the customer's GCP VPC (e.g. via Google Cloud Interconnect / Direct Connect). This connectivity is the customer's responsibility. Harmonic's solution begins at the PSC boundary.

Step-by-Step Procedure
1
Harmonic Deploys LRS Cloud Source Harmonic
Harmonic deploys the LRS Cloud Source VM via Syndicate. The VM must be created with IP forwarding enabled (--can-ip-forward). After the ILB is created, Syndicate adds the ILB frontend IP to the VM's loopback interface and binds the SRT listener to that IP. This means response packets naturally carry src = ILB IP — PSC NAT matches the session without any iptables rules.
▶ How the loopback works click to expand
Why loopback?
GCP's Internal Passthrough NLB forwards packets to the VM with dst = ILB IP unchanged. The kernel would normally drop a packet destined for an IP not assigned to any interface. Adding the ILB IP to the loopback interface (lo) makes the kernel accept it as a local address.

Why bind the listener to the ILB IP?
When a socket is bound to the ILB IP, the OS uses that IP as the source address on all response packets (NAKs, ACKs, retransmissions). PSC NAT tracks sessions expecting src = ILB IP on the return path — binding ensures this is satisfied automatically, with no iptables rules needed.

What --can-ip-forward does
The ILB IP is assigned to the loopback interface (lo), not to ens4 (the NIC). When packets leave through ens4 with src = ILB IP, GCP's network enforces that the source IP must match the NIC's assigned IP. --can-ip-forward disables this check — allowing the ILB IP to be used as source on outgoing packets.

Startup script (on LRS VM)
# Add ILB IP to loopback (applied at VM startup via systemd) ip addr add ILB_IP/32 dev lo
# SRT listener bound to ILB IP (Syndicate passes this at startup) srt-live-transmit "srt://ILB_IP:7086?mode=listener" ...
LRS development requirement: --can-ip-forward must be set at VM creation time in Syndicate. Syndicate must also pass the ILB IP as the SRT listener bind address at startup. This is a one-time Syndicate pipeline change that enables all future PSC-connected Cloud Sources.
Compared to iptables approach: The loopback approach eliminates iptables DNAT/SNAT rules on the VM but requires Syndicate to bind the SRT listener to the ILB IP. The iptables approach requires no application-layer change — the listener stays on 0.0.0.0:7086.
2
Harmonic Creates ILB + PSC Service Attachment Harmonic
For each LRS Cloud Source, Harmonic creates an Internal Load Balancer (ILB) and a PSC Service Attachment. The ILB is the frontend entry point inside the Harmonic VPC. The Service Attachment publishes the ILB as a private service that PSC consumers can connect to.

2. Click Create instance group → New unmanaged instance group

Internal Load Balancer
1. Go to Network Services → Load Balancing → Create load balancer
2. Select Network Load Balancer (TCP/UDP/SSL)
3. Select Passthrough → Internal → Click Continue

3. Name: pfc01-service-attachment

Per-stream resources: Each LRS Cloud Source gets its own ILB, Service Attachment, and PSC NAT subnet. These are independent — a problem with one stream does not affect others.

4
Customer Creates PSC Endpoint Customer
The customer creates a PSC Endpoint (a forwarding rule) in their VPC using the Service Attachment URI from Step 3. GCP assigns a private IP from the customer's own subnet to this endpoint. SRT encoders will connect to this IP.
▶ Console Steps click to expand
1. Go to VPC Network → Private Service Connect → Connected endpoints
2. Click Connect endpoint
3. Target: select Published service
4. Paste the Service Attachment URI from Harmonic
5. Select VPC network and subnet
6. IP address: select Create IP address (GCP assigns from subnet)
7. Click Add endpoint
▶ gcloud CLI click to expand
# Reserve a private IP from customer subnet gcloud compute addresses create pfc01-endpoint-ip \ --region=REGION \ --subnet=CUSTOMER_SUBNET \ --purpose=GCE_ENDPOINT # Create the PSC Endpoint forwarding rule gcloud compute forwarding-rules create pfc01-psc-endpoint \ --region=REGION \ --network=CUSTOMER_VPC \ --address=pfc01-endpoint-ip \ --target-service-attachment=SERVICE_ATTACHMENT_URI # Verify connection is ACCEPTED and get the assigned IP gcloud compute forwarding-rules describe pfc01-psc-endpoint \ --region=REGION \ --format="yaml(IPAddress,pscConnectionStatus)"
Wait for ACCEPTED: The endpoint status must show pscConnectionStatus: ACCEPTED before proceeding. If it shows PENDING, Harmonic needs to verify the customer project is in the Service Attachment accept list.

6
Customer Configures Firewall Rules Customer
The customer creates firewall rules allowing UDP 7086 traffic to and from the PSC subnet. These rules are one-time — they cover all current and future streams on the same subnet, with no changes needed when new streams are added.
▶ Console Steps click to expand
Egress rule (SRT traffic to LRS)
1. Go to VPC Network → Firewall → Create Firewall Rule
2. Name: srt-allow-egress  •  Network: customer VPC
3. Direction: Egress  •  Action: Allow
4. Destination IP ranges: 10.100.0.0/24
5. Protocols and ports: UDP, port 7086
6. Click Create

Ingress rule (SRT return traffic — NAKs, ACKs)
1. Click Create Firewall Rule again
2. Name: srt-allow-ingress  •  Network: customer VPC
3. Direction: Ingress  •  Action: Allow
4. Source IP ranges: 10.100.0.0/24
5. Protocols and ports: UDP, port 7086
6. Click Create
▶ gcloud CLI click to expand
# Allow SRT egress to PSC endpoint subnet gcloud compute firewall-rules create srt-allow-egress \ --network=CUSTOMER_VPC \ --direction=EGRESS \ --allow=udp:7086 \ --destination-ranges=10.100.0.0/24 # Allow SRT return traffic (NAKs, ACKs from LRS) gcloud compute firewall-rules create srt-allow-ingress \ --network=CUSTOMER_VPC \ --direction=INGRESS \ --allow=udp:7086 \ --source-ranges=10.100.0.0/24
SRT is bidirectional UDP: SRT sources are Callers (initiating the connection), but SRT uses UDP which requires both directions to be open. The LRS Listener sends NAKs, ACKs, and retransmitted packets back to the Caller. Both egress and ingress rules are required.

Frequently Asked Questions
▶ Why is --can-ip-forward required for the loopback approach? click to expand
The ILB IP is assigned to the loopback interface (lo), not the VM's NIC (ens4). When the SRT listener sends responses, the packets leave through ens4 with src = ILB IP. GCP's network enforces that packets leaving a NIC must have a source IP matching that NIC's assigned IP. Without --can-ip-forward, GCP blocks these packets and responses carry src = VM NIC IP instead — PSC NAT cannot match the session and drops them. This was confirmed by tcpdump during testing.
▶ Is there NAT involved with PSC? click to expand
From the customer's perspective: no. The PSC Endpoint gets a private IP from the customer's own subnet. SRT encoders connect to this private IP with no internet NAT or public IP involved.

Internally, GCP performs PSC NAT transparently as traffic crosses the VPC boundary. On the Harmonic LRS VM, the loopback approach handles address translation at the listener level — the SRT listener binds directly to the ILB IP, so responses naturally carry the correct source address. This is entirely Harmonic's implementation concern — the customer does not need to configure anything for it.
▶ Does the customer VPC and LRS VPC need to be in the same region? click to expand
Yes. The PSC Endpoint and the Service Attachment must be in the same GCP region. Harmonic deploys the LRS Cloud Source in the same region as the customer's VPC to satisfy this requirement and minimise latency.
▶ Can multiple SRT sources share the same PSC Endpoint? click to expand
No — each SRT stream requires its own PSC Endpoint. Each LRS Cloud Source listens on the same port (7086), so each Cloud Source needs its own private IP — which means its own PSC Endpoint.

For 10 SRT services, the customer creates 10 PSC Endpoints, each connected to its own Harmonic Service Attachment. Each endpoint gets a different IP from the customer's subnet (e.g. 10.100.0.2 through 10.100.0.11).
