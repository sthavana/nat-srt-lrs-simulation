Analysing Harmonic LRS Stream Issues — Azure Log Analytics Guide

A practical reference for investigating packet loss, drops, and stream-health alerts on Harmonic LRS using Azure Log Analytics (the DatapathStats_CL table).

Written as a worked methodology, using a real investigation — Globo, World Cup opening match, 11 June 2026 — as the running example. The investigation changed conclusion three times; each shift was driven by a query result, not a hunch. That is the process working correctly.

Workspace: HarmonicLRSLogAnalytics  •  Table: DatapathStats_CL  •  Query language: KQL (Kusto)

1 — Background: How SRT Works
Most LRS ingest issues come down to understanding SRT, so it's worth a short primer before touching any query. Every concept below has a matching interactive simulation — the links jump straight to the relevant tab.
▶ Open the SRT Protocol simulation — TSBPD, ARQ & Clock Drift

SRT (Secure Reliable Transport) is a UDP-based protocol designed to deliver low-latency video over unreliable networks. Plain UDP has no recovery — if a packet is lost, it's gone. SRT adds a reliability layer on top of UDP without paying the full latency cost of TCP.

Latency buffer. SRT holds packets in a buffer for a configured number of milliseconds (commonly 100 ms or more) before handing them to the decoder. This window is what gives lost packets time to be re-requested and re-delivered. Bigger buffer = more recovery headroom, at the cost of higher end-to-end latency.
▶ See it live — SRT sim › TSBPD Packet Delivery

Loss detection and retransmission. The receiver tracks sequence numbers. When it spots a gap, it sends a NAK (negative acknowledgement) back to the sender asking for that packet again. The sender keeps recently-sent packets in its own buffer so it can honour these requests. This re-request/re-send cycle is the core of SRT's reliability.
▶ See it live — SRT sim › ARQ Retransmission
The recovery race — RTT vs latency buffer. A retransmitted packet is only useful if it arrives before its slot in the latency buffer is played out. This is the single most important relationship in SRT troubleshooting:
If RTT << latency buffer → plenty of time to recover → loss recovered, no impact If RTT ≈ latency buffer → recovery is marginal → some packets arrive too late If RTT > latency buffer → recovery impossible → packets dropped, visible glitches
▶ Try the RTT vs latency slider — SRT sim › ARQ Retransmission (watch the Latency / RTT Ratio)
Loss vs. drop — the critical distinction. These two words mean very different things and the whole investigation hinges on telling them apart:
  • Loss = a packet went missing in transit and triggered a retransmit request. Loss alone is normal and usually invisible — SRT recovers it.
  • Drop = SRT gave up on a packet (couldn't recover it in time, or the buffer moved on). A drop is real, unrecovered loss that can cause a visible artifact.
A stream can show large loss with zero drops and be perfectly healthy. The alert that started our investigation said "packet drop," but the first job was to check whether those were true drops or just recovered loss.
▶ Watch loss become drops — SRT sim › TSBPD (compare "Recovered" vs "Dropped Too Late" vs "Effective Loss")
Primary / backup legs. LRS streams are often configured with redundant SRT paths — a primary and a backup leg carrying the same content over different paths. If one leg degrades, the other carries the stream. A single degraded leg is not a stream outage if its partner stayed healthy.
2 — The Data Source: DatapathStats_CL
All analysis below queries the DatapathStats_CL table in the HarmonicLRSLogAnalytics workspace. Each row is a periodic stats sample from one socket (one leg) of one stream.

Important — what "send" and "receive" mean here. The snd_* and recv_* prefixes are relative to the endpoint reporting the stat, not fixed roles for the encoder and the LRS VM. The same box can be a receiver on one socket and a sender on another (SRT is bidirectional, and LRS may forward a stream onward). The endpoint's actual role is given by id_s (e.g. ingress-...) and upstream_role_s, not the send/receive prefix. On an ingress endpoint, recv_* is the encoder→LRS feed — the primary signal for ingest health — while snd_* reflects that endpoint's own outbound/return traffic (control packets and/or onward forwarding).
▶ Send-side metrics (data this endpoint sent) click to expand

recv_byte_drealTotal bytes received.

FieldTypeMeaning

The four fields that actually answer "is there a problem?"
  1. recv_pkt_dropped_d — unrecovered loss at the receiving endpoint. On ingress this is the encoder→LRS feed, so non-zero here is the clearest sign "the LRS didn't get the full feed in time."
  2. snd_pkt_dropped_d — unrecovered loss on this endpoint's outbound side.
  3. ms_rtt_d vs ms_latency_d — whether recovery was even possible.
  4. upstream_role_s — whether a healthy redundant leg was carrying the stream.
Loss counters (*_loss_d) tell you something happened; drop counters tell you whether it mattered.
3 — The Investigation Methodology
Guiding principle: let the data eliminate variables, and never state a conclusion the data hasn't earned.
  1. Establish what "normal" looks like (baseline)
  2. Confirm the alert is real (loss vs. drop, and as a %)
  3. Find WHEN it happened (timeline)
  4. Find WHICH streams were affected
  5. Find the SHARED component among affected streams
  6. Eliminate components the data rules out
  7. State only what the data proves; flag the rest as open
4 — The Queries: What We Ran and Why
Each query is shown with the reasoning that prompted it and what its result told us — the actual sequence from the Globo investigation.
1
Loss over time, per stream Baseline
Why: The alert was "packet drops during the match." First question — did all streams drop together (shared cause) or independently (per-stream cause)?
DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize SendLoss = sum(snd_pkt_loss_d), RecvLoss = sum(recv_pkt_loss_d) by bin(TimeGenerated, 1m), name_s | render timechart
Result Loss looked scattered and small across the evening — no obvious synchronised spike. (This later proved misleading because raw counts hide the percentage and we were looking at the wrong window — a lesson in itself.)
2
Bitrate over time Rule out surge
Why: To test the "match traffic spike saturated a shared pipe" hypothesis. If bitrate was flat, there was no surge to blame.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize AvgSendBitrate = avg(snd_bitrate_d) by bin(TimeGenerated, 1m), name_s | render timechart
Result Rock-steady ~33 Mbps throughout. This killed the bitrate-spike / pipe-saturation theory — there was no surge.
3
Latency over time Rule out congestion
Why: Congestion shows up as latency climbing before loss. Flat latency = no building congestion.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize AvgLatency = avg(ms_latency_d), MaxLatency = max(ms_latency_d) by bin(TimeGenerated, 1m) | render timechart
Result Latency flat (buffer at 100 ms, RTT tiny). No congestion signature.
4
Loss as a percentage, per stream Normalise
Why: Raw loss counts are meaningless without volume. 18 lost out of 50 million is nothing. This converts counts to the only metric that reflects quality.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize TotalSent = sum(snd_pkt_d), TotalSendLoss = sum(snd_pkt_loss_d) by name_s | extend SendLossPct = round(100.0 * TotalSendLoss / TotalSent, 4) | sort by SendLossPct desc
Result Worst send-loss was 0.0054%. This looked like an all-clear — but it only measured send-side loss, not receive-side drops, which is what tripped us up next.
5
Recovery proof (loss vs. retransmit vs. drop) Turning point
Why: To confirm whether loss was actually recovered. This requires the drop counters, not just loss.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | summarize TotalLoss = sum(snd_pkt_loss_d), TotalRetrans = sum(snd_pkt_retrans_d), TotalSendDropped = sum(snd_pkt_dropped_d), TotalRecvDropped = sum(recv_pkt_dropped_d) by name_s | sort by TotalLoss desc
Result This overturned the all-clear. LL06 showed 23,267 receive-dropped and LL07 showed 14,056 — genuine unrecovered loss, despite near-zero loss percentages on the send side. Lesson: always check recv_pkt_dropped_d, not just loss.
6
Finding when the drops happened Timeline
Why: To locate the drops in time and see if they clustered. Filter to rows where a drop actually occurred.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where name_s has "LL06" or name_s has "LL07" or name_s has "LL08" | where TimeGenerated between (datetime(2026-06-11T18:00:00Z) .. datetime(2026-06-11T23:00:00Z)) | where snd_pkt_dropped_d > 0 or recv_pkt_dropped_d > 0 | project TimeGenerated, name_s, snd_pkt_loss_d, snd_pkt_dropped_d, recv_pkt_loss_d, recv_pkt_dropped_d, ms_rtt_d | sort by TimeGenerated asc
Result All drops were concentrated in a ~3-minute window, 18:25–18:28 UTC — about 35 minutes before kickoff. This destroyed the "match-load" theory entirely; the event was pre-match.
7
Magnitude and scope of the window Quantify
Why: Quantify the drop as a percentage within the tight window, and confirm it was only these three streams.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where name_s has "LL06" or name_s has "LL07" or name_s has "LL08" | where TimeGenerated between (datetime(2026-06-11T18:24:00Z) .. datetime(2026-06-11T18:30:00Z)) | summarize RecvPkts = sum(recv_pkt_d), RecvDropped = sum(recv_pkt_dropped_d), SndDropped = sum(snd_pkt_dropped_d) by name_s | extend RecvDropPct = round(100.0 * RecvDropped / RecvPkts, 4)
Result LL06 = 1.07% unrecovered, LL07 = 0.64% unrecovered. These are real numbers that can cause visible artifacts — no longer dismissible as micro-loss.
8
Per-second shape and the two-leg discovery Key discovery
Why: Was it one big burst or sustained? And recv_bitrate_d per row revealed something unexpected.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where name_s has "LL06" | where TimeGenerated between (datetime(2026-06-11T18:24:00Z) .. datetime(2026-06-11T18:30:00Z)) | project TimeGenerated, recv_pkt_d, recv_pkt_loss_d, recv_pkt_dropped_d, recv_bitrate_d, ms_rtt_d, link_bw_d | sort by TimeGenerated asc
Result Two rows per timestamp — one healthy (33 Mbps, 0 loss) and one degraded (17 Mbps, heavy loss). This revealed the stream had two legs, and only one degraded. (link_bw_d was empty — abandoned that column.)
9
Identifying the shared component Correlate
Why: To find what LL06/LL07/LL08 had in common that the unaffected streams didn't.
DatapathStats_CL | where account_prefix_s has "globocomuni" | where name_s in ("LL06 - PSC", "LL07 - PSC", "LL08 - PSC") | where TimeGenerated between (datetime(2026-06-11T18:24:00Z) .. datetime(2026-06-11T18:30:00Z)) | distinct name_s, id_s, upstream_pipeid_s, upstream_role_s, region_s, socket_d, platform_s | sort by name_s asc
Result Separate ingress IDs (independent streams), but all in zone southamerica-east1-b, and each had primary + backup roles — confirming the two-leg redundancy structure.
10
Redundancy verification Decisive query
Why: The final question — for each stream, was a healthy leg always available, or did primary AND backup fail together?
DatapathStats_CL | where account_prefix_s has "globocomuni" | where name_s in ("LL06 - PSC", "LL07 - PSC", "LL08 - PSC") | where TimeGenerated between (datetime(2026-06-11T18:24:00Z) .. datetime(2026-06-11T18:30:00Z)) | project TimeGenerated, name_s, upstream_role_s, socket_d, recv_pkt_loss_d, recv_pkt_dropped_d, recv_bitrate_d | sort by name_s asc, TimeGenerated asc
Result
  • LL06 — only the backup leg degraded; primary clean throughout → protected.
  • LL07 — only the primary leg degraded; backup clean throughout → protected.
  • LL08 — both legs took minor loss but recv_pkt_dropped_d = 0 → fully recovered.
In every case a healthy path carried the stream. Redundancy worked as designed; no broadcast impact.
5 — How to Narrow Down and Eliminate Variables
The investigation worked by progressively ruling components out. This is the reusable logic:
QuestionHow to answer it
Is the loss real or recovered?Check recv_pkt_dropped_d and snd_pkt_dropped_d. If both are ~zero, the loss was recovered and there is no impact — regardless of how big the loss counters look. Only drops indicate possible viewer impact.
Is it capacity / congestion?Look at snd_bitrate_d / recv_bitrate_d (flat = no surge) and the ms_latency_d trend (flat = no building congestion). Steady bitrate + non-climbing latency rules out pipe saturation.
Could recovery even succeed?Compare ms_rtt_d to ms_latency_d. If RTT is a small fraction of the buffer, recovery had ample headroom and late-arrival drops are unlikely to be the cause.
Together or independently?Group drops by bin(TimeGenerated, 1m) and name_s. Clustering in time points to a common trigger; scattered points to per-stream causes.
What do affected streams share?Use distinct on region_s, id_s, upstream_pipeid_s, socket_d. A shared region/pipe/endpoint that the unaffected streams don't share is your candidate root cause.
Was redundancy protective?Split by upstream_role_s. If for each stream at least one leg (primary or backup) stayed clean at all times, the stream was protected.
Eliminate, don't assume. The single biggest discipline: a correlation (e.g. "all in one zone") is not a cause. We could not blame the GCP zone, because the healthy legs in that same zone had zero loss — a zone-wide fault would have hit them too.
6 — Reaching a Defensible Conclusion
State what the data proves. Flag what it doesn't. The Globo investigation landed at:
Proven:
  • A ~3-minute degradation at 18:25–18:28 UTC (15:25–15:28 São Paulo), ~35 min before kickoff.
  • Affected one leg each of LL06, LL07, LL08 — all in southamerica-east1-b.
  • LL06 ~1.07% and LL07 ~0.64% unrecovered loss on the degraded leg only.
  • In every case the redundant leg stayed healthy → no broadcast impact.
Not proven (open / would need more data):
  • The specific faulty component. Receiver-side stats show loss happened in transit but not which hop. The healthy same-zone legs make a blanket "GCP zone fault" unlikely.
  • To attribute the cause, you'd need data from other points in the chain at that exact window: Globo encoder send logs, Direct Connect / Interconnect metrics, GCP zone status history, or PSC / ILB metrics.
The headline: the alert was real, but redundancy absorbed it — the streams were protected, and the broadcast was not affected.
7 — Reusable Query Snippets
A quick-reference toolkit for the next investigation. Replace the account prefix, stream names, and time window.
Health summary across all streams (run this first):
DatapathStats_CL | where account_prefix_s has "<account>" | where TimeGenerated between (datetime(<start>) .. datetime(<end>)) | summarize Sent = sum(snd_pkt_d), Recv = sum(recv_pkt_d), SendDropped = sum(snd_pkt_dropped_d), RecvDropped = sum(recv_pkt_dropped_d) by name_s | extend RecvDropPct = round(100.0 * RecvDropped / Recv, 4), SendDropPct = round(100.0 * SendDropped / Sent, 4) | sort by RecvDropPct desc
List every column in the table (when a field name errors):
DatapathStats_CL | getschema | project ColumnName, ColumnType | sort by ColumnName asc
Find exactly when drops occurred:
DatapathStats_CL | where account_prefix_s has "<account>" | where snd_pkt_dropped_d > 0 or recv_pkt_dropped_d > 0 | where TimeGenerated between (datetime(<start>) .. datetime(<end>)) | project TimeGenerated, name_s, upstream_role_s, recv_pkt_dropped_d, snd_pkt_dropped_d | sort by TimeGenerated asc
Check redundancy (per-leg health):
DatapathStats_CL | where account_prefix_s has "<account>" | where name_s == "<stream>" | where TimeGenerated between (datetime(<start>) .. datetime(<end>)) | project TimeGenerated, upstream_role_s, socket_d, recv_pkt_loss_d, recv_pkt_dropped_d, recv_bitrate_d | sort by TimeGenerated asc
Appendix — Time Zone Reference
LRS timestamps in Log Analytics are UTC. São Paulo (Globo) is UTC−3 year-round (no DST). When correlating with broadcast schedules, convert:
UTC − 3 hours = São Paulo (BRT) e.g. 18:25 UTC = 15:25 São Paulo
Always quote both UTC and local time when reporting to a customer, to avoid cross-reference confusion.
