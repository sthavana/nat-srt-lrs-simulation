▶ VOS 360 / LRS — PSC (Private Service Connect) Loopback & iptables click to expand
How SRT encoders connect to Harmonic LRS Cloud Source VMs on GCP via PSC (Private Service Connect). Two validated approaches: PF01 (iptables) uses DNAT+SNAT rules so the listener stays on 0.0.0.0:7086 with no application change. PF02 (loopback) adds the ILB IP to the lo interface and binds the SRT listener directly to that IP — no iptables needed. Both require --can-ip-forward=True on the LRS VM so GCP allows response packets with src=ILB IP.

▶ SNAT / DNAT — PSC + ILB Return Path click to expand
The root cause is the Internal Passthrough NLB (ILB). Unlike a proxy load balancer, it does not terminate the connection — it forwards packets straight to the backend VM with dst = ILB frontend IP (172.18.64.86), not the VM's own IP (172.18.64.93). The Linux kernel on the LRS VM gets a packet for an address it doesn't own and silently drops it.

DNAT (PREROUTING): fires before the kernel routing decision. Rewrites dst 172.18.64.86 → 172.18.64.93 so the kernel delivers the packet locally to the SRT listener.

SNAT (POSTROUTING): fires as the response leaves the VM. Rewrites src 172.18.64.93 → 172.18.64.86 so PSC NAT recognises the return traffic — it was tracking a session keyed to 172.18.64.86, so a response from 172.18.64.93 is unknown to it and gets silently dropped.

Also notice the asymmetric return path: the response from the LRS VM goes directly to the PSC NAT subnet — the ILB is not in the return path.
